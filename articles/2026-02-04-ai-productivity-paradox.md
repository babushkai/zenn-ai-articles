## Part 1: METR研究の衝撃

### 研究の概要

METRは**ランダム化比較試験（RCT）**を実施しました。

これは医学研究でも使われる、**最も信頼性の高い実験手法**です。

```
📊 METR研究の設計:
├── 参加者: 16人のベテランOSS開発者
├── タスク数: 246の実際のコーディングタスク
├── コードベース: 平均100万行以上の成熟リポジトリ
├── 経験: 平均5年、1,500コミット以上
└── ツール: Cursor Pro + Claude 3.5/3.7 Sonnet
```

### 衝撃の結果

```
📈 生産性への影響:

AI使用時: ████████████████████████████ +19%遅くなった
AI不使用: ████████████████████ ベースライン

開発者の感想: -20%速くなったと感じた
実際の結果:   +19%遅くなった

↓
予想と現実のギャップ: 43ポイント
```

**開発者は「速くなった」と感じているのに、実際には遅くなっている。**

**全員、大きく外れました。**

## Part 2: なぜAIで遅くなるのか？

### 「隠れたコスト」の存在

AIコーディングには、**見えないオーバーヘッド**があります。

```
⏱️ AI使用時の追加コスト:

1. プロンプト作成時間
   └── 「何を聞くか」を考える時間

2. 出力の検証時間
   └── AIが正しいか確認する時間

3. コードレビュー時間
   └── 生成されたコードを読む時間

4. 修正時間
   └── 間違いを直す時間

5. コンテキスト切り替え
   └── 自分の思考 ↔ AIの出力
```

**手動コーディングでは、これらのコストは存在しません。**

### 受け入れ率は44%未満

METR研究によると：

- **AI生成コードの受け入れ率: 44%未満**
- **75%の開発者**: 「AIの出力を全行読んでいる」
- **56%の開発者**: 「大幅な修正を加えている」

つまり、AIは「コードを書いてくれる」のではなく、「**レビューが必要な下書きを量産する**」のです。

### ベテランほど遅くなる理由

```
👨‍💻 経験者がAIで遅くなる理由:

1. 既にコードベースを熟知している
   └── AI説明のオーバーヘッドが無駄

2. 自分のスタイルがある
   └── AIコードを自分流に直す時間

3. 品質基準が高い
   └── AIの出力を徹底的にレビュー

4. 問題解決パターンを知っている
   └── AIに聞くより自分で書く方が速い
```

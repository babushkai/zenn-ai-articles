## 「コードを読まずにアプリを作る」——あなたはこれを許せますか？

2025年、開発者界隈を二分する言葉が生まれました。

**Vibe Coding（バイブコーディング）**。

「AIに丸投げして、なんとなく動けばOK」——そんな開発スタイルが、Collins英語辞典の**2025年Word of the Year**に選ばれたのです。

そして今、2026年1月。この「Vibe Coding」をめぐって、開発者たちの間で**激しい論争**が巻き起こっています。

## 結論から言うと...

:::message alert
**Vibe Codingは「使い方次第」で天国にも地獄にもなる。**
問題は、多くの人が「地獄の使い方」をしていることだ。
:::

## そもそもVibe Codingとは？

OpenAIの共同創業者であり、TeslaのAI責任者だったAndrej Karpathyが2025年2月に提唱した概念です。

**定義**：
> LLMにプロジェクトを説明し、生成されたコードを**レビューも編集もせずに**、実行結果だけを見て評価し、改善を依頼するスタイル

つまり：

```
従来の開発:
1. 要件定義 → 2. 設計 → 3. コーディング → 4. レビュー → 5. テスト → 6. デプロイ

Vibe Coding:
1. AIに「〇〇作って」 → 2. 動いた！ → 3. デプロイ
```

...おい、2〜5どこいった？

## 賛成派の主張：「10倍速い」は本当だった

実際、Vibe Codingの生産性は凄まじい。

### 実例：10分でアプリ完成

Stack Overflowの記事によると、ある開発者はVibe Codingツールを使って**わずか10分**で「Redditのトイレレビューアプリ」を作成しました。

通常なら数時間〜数日かかる作業です。

### 学習効率10倍

物理学者の友人がLLMで開発を始めたところ、「**学習効率が10倍になった**」と報告。バグの説明を即座に得られ、情報収集が劇的に速くなったとのこと。

### 非エンジニアの参入障壁消滅

プログラミング経験ゼロの人でも、自然言語でアプリを作れる時代に。これは**民主化**と言えるかもしれない。

## 反対派の主張：「開発地獄」の始まり

しかし、プロの開発者たちは警鐘を鳴らしています。

### 衝撃の調査結果（2026年1月）

UC San DiegoとCornell大学の研究（2026年1月5日発表）によると：

> **経験豊富なソフトウェア開発者は、Vibe Codingアプローチを採用していない**

AIコーディングツールが普及しても、プロは「バイブ」に頼っていないのです。

### 66%が「フラストレーション」を経験

AIが生成するコードは「**ほぼ正しいが、完全には正しくない**」。

この「あと一歩」の修正に、結局ベテラン開発者の時間が取られる。これを**「生産性税」**と呼ぶ声も。

### 恐怖の実例：コードベースの3分の1が重複

ある開発者の告白：

> 「本番環境のコードの**1/3以上が重複**していた。あるコンポーネントは推奨サイズの**9倍**に膨れ上がり、環境がクラッシュ。同じテキスト正規化関数が**15個のファイルに散在**していた」

AIは同じ問題を毎回違う方法で解決するため、コードベースが**パッチワークの怪物**と化すのです。

### セキュリティ脆弱性の量産

2025年5月、スウェーデンのVibe CodingアプリLovableで深刻な問題が発覚：

> **1,645個のアプリのうち170個（約10%）に個人情報が漏洩する脆弱性**

住所、メールアドレス、パスワード...本番で扱うべきでないデータが、誰でもアクセス可能な状態に。

## Linus Torvaldsも使っていた！？

驚くべきことに、Linuxの生みの親Linus Torvalds本人が、2026年1月にVibe Codingを使ったことを認めました。

AudioNoiseプロジェクトのPython可視化ツールについて：

> 「このPython可視化ツールは、基本的にVibe Codingで書かれた」

ただし、これは**個人プロジェクトの一部**であり、Linuxカーネル本体ではありません。

## 「Vibe Coding」vs「AIアシスト開発」：決定的な違い

ここで重要な区別があります。

| | Vibe Coding | AIアシスト開発 |
|--|-------------|--------------|
| コードレビュー | しない | する |
| 理解度 | 動けばOK | 理解して使う |
| テスト | 実行結果のみ | 単体・統合テスト |
| 責任 | AI任せ | 人間が持つ |
| 適用範囲 | プロトタイプ | 本番システムも可 |

Andrew Ng（DeepLearning.AI創設者）はこう批判しています：

> 「『Vibe Coding』という言葉は誤解を招く。ソフトウェアエンジニアがAIツールを使うとき、**単に『バイブ』に従っているわけではない**」

## プロが実践すべき「ハイブリッドアプローチ」

では、AIの恩恵を受けつつ、品質を保つにはどうすれば良いのか？

### 1. Vibe Codingは「プロトタイプ限定」

```
OK: MVPの検証、個人プロジェクト、学習目的
NG: 本番システム、顧客データを扱うアプリ、医療・金融
```

### 2. 「AIと一緒に」コーディング

あるベテラン開発者の言葉：

> 「私はエージェントと**一緒に**コーディングするのが好きだ。Vibe Codingではなく、**協働**として」

AIに完全委任するのではなく、**ペアプログラミングの相棒**として使う。

### 3. レビューは必須

AIが生成したコードを、そのまま本番に入れない。最低限：

- [ ] セキュリティチェック
- [ ] 重複コードの確認
- [ ] テストカバレッジ
- [ ] パフォーマンス検証

### 4. Claude Codeでの実践例

```markdown
# CLAUDE.mdに追加すべきルール

## コーディングルール
- AIが生成したコードは必ずレビューすること
- セキュリティに関わる部分は人間が確認
- テストなしのマージは禁止
- 重複コードは即リファクタリング
```

## 若手開発者への警告

最も懸念されているのは、**スキルギャップの拡大**です。

> 「若手開発者は自分でコードを書く能力が低下し、Vibe Codingツールへの依存度が高まるだろう。これはジュニアとシニアの間のスキルギャップを広げる」

AIがすべてを書いてくれる時代に、**なぜ自分で学ぶ必要があるのか？**

答えは明白です：

1. **AIのミスを見つけるには、正解を知る必要がある**
2. **本番障害時、AIは責任を取らない**
3. **AIの提案を評価するには、基礎知識が必須**

## まとめ：2026年、開発者に求められる姿勢

:::message
**Vibe Codingは「ツール」であり「スタイル」ではない**
:::

### 使うべき場面
- プロトタイピング
- 学習・実験
- 個人プロジェクト
- アイデア検証

### 避けるべき場面
- 本番システム（レビューなしでは）
- 顧客データを扱うアプリ
- 医療・金融・セキュリティ重要領域
- チーム開発のメインワークフロー

### 2026年の開発者に必要なスキル

1. **AIの出力を評価できる目** - 良いコードと悪いコードを見分ける
2. **セキュリティの基礎知識** - 脆弱性を検出できる
3. **アーキテクチャ設計力** - AIに正しい指示を出せる
4. **コードレビュー力** - AIの「ほぼ正しい」を「完全に正しい」にできる

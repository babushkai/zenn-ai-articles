**AIにコードを書かせまくった結果、何が起きたか。**

2025年、「Vibe Coding」という言葉が流行りました。

> **Vibe Coding**: AIに雰囲気（vibe）だけ伝えてコードを生成させる開発スタイル

そして2026年、その**代償**が明らかになりつつあります。

:::note alert
**「AIスロップ」問題**: AIが生成した低品質コードが本番コードベースに蓄積し、技術的負債が爆発的に増加している。
:::

## 何が起きているのか

### AIスロップの実態

```
「AIスロップ」の特徴
────────────────────────────────────────
❌ 動くけど誰も理解できないコード
❌ 過剰に抽象化された設計
❌ 意味不明なコメント
❌ コピペの痕跡だらけ
❌ テストは通るが本質的にバグってる
❌ パフォーマンスが謎に悪い
────────────────────────────────────────
```

### 実際の事例

```python
# AIが生成した「動くけど意味不明な」コード

def process_data(data):
    """Process the data."""  # AIの典型的な無意味コメント
    result = []
    for item in data:
        if item is not None:  # なぜNoneチェック？
            temp = item
            if temp:  # 二重チェック？
                processed = temp  # 何も処理してない
                if processed is not None:  # また？
                    result.append(processed)
    return result if result else []  # 空リストを空リストで返す？

# 人間が書いたら:
def process_data(data):
    return [item for item in data if item]
```

## なぜこうなったのか

```
┌─────────────────────────────────────────────────────────┐

```

## BYOK移行トレンド

### パワーユーザーの反乱

```
CursorからClaude Code/OpenCodeへの移行理由
────────────────────────────────────────
1. API キーを直接管理したい（コスト把握）
2. プロンプトを完全にコントロールしたい
3. 「おまかせ」ではなく「指示」したい
4. 生成されたコードを必ず理解してからマージ
────────────────────────────────────────
```

### BYOK（Bring Your Own Key）とは

| 方式 | 特徴 |
|------|------|
| **SaaS型**（Cursor等） | 便利だが「ブラックボックス」 |
| **BYOK型**（Claude Code等） | 面倒だが「完全制御」 |

## 解決策：責任あるAIコーディング

### 新しいベストプラクティス

```
責任あるAIコーディングの原則
────────────────────────────────────────
✅ AIの出力は「下書き」として扱う
✅ すべてのAI生成コードを人間がレビュー
✅ 「なぜこう書いたか」を説明できないコードはマージしない
✅ AIに「リファクタ」させる前に、まず理解する
✅ テストだけでなく、可読性もチェック
────────────────────────────────────────
```

### コードレビューチェックリスト

```markdown
## AI生成コードレビューチェックリスト

### 理解度
- [ ] コードの意図を説明できる
- [ ] なぜこの設計なのか理解している
- [ ] エッジケースを把握している

### 品質
- [ ] 過剰な抽象化がない
- [ ] 不要なnullチェックがない
- [ ] コメントが意味のある内容

### 保守性
- [ ] 6ヶ月後の自分が理解できる
- [ ] チームメンバーが理解できる
- [ ] デバッグ可能
```

### 組織的対策

| 対策 | 内容 |
|------|------|
| **AI コード比率上限** | 1PR あたり50%まで等 |
| **必須レビュー** | AI生成コードは2人以上でレビュー |
| **定期的な棚卸し** | AI生成コードの品質監査 |
| **教育** | 「AIを使う技術」のトレーニング |

## AIコーディングの正しい使い方

### やるべきこと

```python
# ✅ 良い使い方: AIに「叩き台」を作らせる

# 人間: 「ユーザー認証のミドルウェアを作って」
# AI: （コード生成）
# 人間: （理解した上で）修正・改善・マージ
```

### やってはいけないこと

```python
# ❌ 悪い使い方: AIに「丸投げ」

# 人間: 「なんかいい感じにして」
# AI: （コード生成）
# 人間: 「動いた！マージ！」
#
# 3ヶ月後: 「このコード誰が書いた？意味わからん」
```

## 企業への警告

### 技術的負債の時限爆弾

```
AIスロップによる損害（試算）
────────────────────────────────────────
中規模スタートアップ（エンジニア20人）の場合:

• AI生成コード比率: 60%
• うち「スロップ」: 30%
• 年間コード行数: 50万行

→ 毎年1.4億円の「隠れ負債」が蓄積
────────────────────────────────────────
```
